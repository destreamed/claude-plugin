## Phase 1 — Protocol footnote infrastructure

**Repo:** `destreamed-backend`. Implements the [`_meta.destreamed`](./protocol-presence-footnote.md) envelope server-side.

| Step | Description |
|---|---|
| P1.1 | Slim footnote: emit `protocol_version`, `server_time`, `presence_etag`, `advisories: []`, `workflow_state` on every API/MCP response |
| P1.2 | Presence tracking: Redis-backed map `(stream_id) → [(agent_token, last_seen, activity, focus)]`, 60s TTL window |
| P1.3 | Rich footnote: emit `presence` and `context_drift` on focus/list endpoints (per protocol classification table) |
| P1.4 | ETag computation + `If-Destreamed-Presence-None-Match` request header support; on match, omit `presence`/`context_drift` and set `presence_unchanged: true` |
| P1.5 | Initial advisory codes: `stream_freeze_active`, `agent_token_expiring`, `agent_token_expired`, `rate_limit_close`, `oauth_scope_insufficient`, `peer_tuner_joining` — emission rules per code |
| P1.6 | `/.well-known/destreamed-protocol` endpoint announces supported versions |

**Acceptance:** every `200`/`4xx`/`5xx` response carries `_meta.destreamed`; presence mirror in Redis updates correctly; etag-skip path saves ≥50% bytes on repeat calls in steady state (measured).

**Approval echo body:** same template (What/Why/Impact).

## Phase 2 — Agent registration

**Repo:** `destreamed-backend`.

| Step | Description |
|---|---|
| P2.1 | `POST /api/v1/agents/register` endpoint with body validation (tuner_slug, client_kind, client_version, session_uuid, capabilities) |
| P2.2 | Agent-token middleware: validate `X-Destreamed-Agent-Token` on every authenticated request; set request-scoped actor |
| P2.3 | `POST /api/v1/agents/refresh` endpoint, refresh window = last 25% of TTL |
| P2.4 | OAuth scope additions: `tuners:read`, `echoes:approve` registered in OAuth metadata |

**Acceptance:** plugin (or curl test) can register, carry token, get presence-aware footnotes, refresh before expiry, re-register after expiry without user re-auth.

## Phase 3 — Plugin client foundations

**Repo:** `destreamed-claude-plugin`. Branch: `v0.4`.

| Step | Description |
|---|---|
| P3.1 | Branch `v0.4` cut from main; `plugin.json` version → `0.4.0-pre.1` |

## Phase 4 — Focus mechanic

**Repo:** `destreamed-claude-plugin`.

| Step | Description |
|---|---|
| P4.1 | Slash command `/dt <task-id>` — parses `DESTREAM-81`-format, calls `tasks/:id/context`, primes Claude with tuner+memos+echoes |

**Acceptance:** end-to-end `/dt DESTREAM-81` works; presence in chat shows other actors if they're in the same stream.

## Phase 5 — Approval flow

**Repo:** `destreamed-claude-plugin`.

| Step | Description |
|---|---|
| P5.1 | `destreamed-propose` skill updated to use new approve/deny endpoints and the workflow_state footnote |
| P5.2 | `/dt approve <echo-id>` and `/dt deny <echo-id> [reason]` slash commands |
| P5.3 | Wait-loop: subscribe via SSE if available, fallback polling (5s/30s/2min), stop on workflow_state.current_gate change |
| P5.4 | `/dt approve --here <echo-id>` chat-override (option **b** from earlier decision); audit-trail field `approved_via: "chat-override"` set |
| P5.5 | Decision-outcome handlers: APPROVE executes, DENY pivots, MODIFY treats counter as new direction |

**Acceptance:** propose → wait → approve via UI → tuner resumes; same flow with `/dt approve` shortcut works. Denial pivots cleanly.

## Dependency graph

```
Phase 0 (Phase-E backend)
   └── Phase 1 (footnote infra) ── Phase 2 (agent registration)
                                       └── Phase 3 (plugin foundations)
                                              └── Phase 4 (focus mechanic)
                                                     └── Phase 5 (approval flow)
                                                            └── Phase 6 (subagents)
                                                                   └── Phase 7 (release)
```

Strict serial dependencies. Phase 0 can start tomorrow regardless of plugin work; everything client-side blocks until Phase 0+1+2 are deployable on a staging endpoint. We can shorten the critical path by overlapping client scaffolding (P3.1) once Phase 1's slim footnote is shipped, even before agent registration is done — agent_token can be stubbed.

## What this plan deliberately does NOT do

- It does not start Phase E backend work — that's a Destreamed-team decision (cf. open question in `v0.4-spec.md`).
- It does not pre-register Claude-Code-specific behavior in Destreamed's general API (multi-client parity).
- It does not yet handle workflow primitives beyond `human-gate` — those are v0.5+ per spec.
- It does not block on perfect OAuth-scope ergonomics; if `tuners:read` requires a re-auth, that's documented in v0.4 release notes.
