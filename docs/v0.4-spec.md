# v0.4 Spec — Task-Focus Engine

> Status: **DRAFT**. Single source of truth for the v0.4 redesign. No code lands until this doc is signed off.

## TL;DR

A Claude Code session is no longer scoped to a project — it's scoped to a **Destreamed task**. One slash command (`/dt DESTREAM-81`) loads everything Claude needs to act competently in that task's context: the assigned tuner's persona, the stream's memos, the task's prior echoes, and any linked beats. Switching to another task in the same session swaps the entire context cleanly.

## Goals

1. **Task-first, not project-first.** Sessions begin by picking a task. Generic "open Claude in this folder" starts are no longer the default.
2. **Persona-as-tuner.** Each task has an `assigned_tuner`. Loading the task installs that tuner's persona — system-prompt layer, tool whitelist, default memos.
3. **Hot-swap.** `/dt <new-id>` mid-session replaces persona + memos + echoes-context. The active conversation history stays, but the *operating context* turns over.
4. **Subagents stay in role.** Spawned subagents inherit the parent's tuner profile + memos + task context, with a `needs_approval` gate on every material action.
5. **Sharing via Destreamed.** Tuners, memos, and tasks are server-side. Anyone with stream membership gets the same setup with no plugin update.

## Concepts

### Task ID format

`<STREAM-CODE>-<NUMBER>`, e.g., `DESTREAM-81`. Auto-generated by Destreamed when a task is created. Stream code is the uppercased prefix derived from the stream slug. This ID is the canonical handle for `/dt` switches and for deep-links.

### Tuner profile (server-side)

A tuner is a server-managed record:

| Field | Type | Purpose |
|---|---|---|
| `id` | uuid | Stable identifier |
| `slug` | string | Human reference (e.g., `backend-engineer`) |
| `display_name` | string | Shown in pickers and focus banner |
| `system_prompt` | string | The persona spec — added as a system-message layer when this tuner is active |
| `allowed_tools` | string[] | MCP/Claude tool whitelist (advisory; enforcement is via system prompt) |
| `default_memos` | uuid[] | Memos every task assigned to this tuner inherits |
| `model_preference` | enum? | `opus` / `sonnet` / `haiku` / null (Claude Code may override) |

When a task is fetched, its `assigned_tuner` field returns the full tuner profile inline.

### Task context bundle

One API call, one round trip, everything to switch focus:

```json
{
  "task": {
    "id": "DESTREAM-81",
    "title": "Fix race condition in stream-invite accept",
    "body": "...",
    "status": "in_progress",
    "stream_id": "..."
  },
  "stream": {
    "id": "...",
    "slug": "destreamed-backend",
    "name": "Destreamed Backend"
  },
  "tuner": {
    "slug": "backend-engineer",
    "display_name": "Backend Engineer",
    "system_prompt": "You are a senior Rust/Axum engineer...",
    "allowed_tools": ["read", "write", "bash:cargo", "mcp__destreamed__*"],
    "model_preference": "opus"
  },
  "memos": [
    { "id": "...", "title": "Rust style", "body": "..." },
    { "id": "...", "title": "No mocks in integration tests", "body": "..." }
  ],
  "echoes": [
    {
      "id": "...",
      "kind": "echo",
      "body": "Found root cause in services/streams.rs:142",
      "needs_approval": false,
      "approved": null,
      "created_at": "2026-04-29T..."
    }
  ],
  "linked": [
    { "id": "...", "kind": "memo", "title": "Stream invite spec" },
    { "id": "...", "kind": "question", "title": "Backfill or unique index?", "open": true }
  ]
}
```

### Subagent inheritance

When the main thread spawns a subagent (Claude Code's `Agent` tool), the prompt template is built from:

- The active tuner's `system_prompt` (verbatim)
- A condensed memo summary (titles + 1-line gist each, full bodies on request)
- The active task's title + body
- The specific sub-instruction from the parent
- A reminder line: "needs_approval: true is mandatory for any material change. Echo back to task DESTREAM-81 on completion."

The subagent's tool whitelist matches the parent tuner's `allowed_tools`. It can read everything, but writes (file edits, beat drops, completion) require a `needs_approval` gate.

## User experience

### Switching focus

```text
$ claude
> /dt DESTREAM-81
🎯 Switching focus → DESTREAM-81 — "Fix race condition in stream-invite accept"
   Tuner:  backend-engineer
   Stream: destreamed-backend
   Memos:  4 loaded (Rust style, error handling, no-mocks-in-tests, migration-naming)
   Echoes: 3 prior (last: "Found root cause in services/streams.rs:142")
   Linked: 1 spec memo, 1 open question

   ⚠ Open question on this task:
     "Do we want to backfill existing rows, or is the unique index enough?"
     → answer with `/dt answer <text>` or address it in your plan.

   Ready.
```

Without arguments, `/dt` opens an interactive picker over the user's open tasks (fuzzy-filterable by title and stream).

### Subagent

```text
> Plan was approved. Spawn a subagent to write the integration test for this fix
  while I refactor the handler.

[main] spawning subagent (tuner: backend-engineer, scope: write integration test)
       (subagent inherits memos, gets needs_approval gate on production writes)
```

When the subagent finishes, it posts a single echo on `DESTREAM-81` summarizing the work, and returns control.

### Approval

Material steps (`add_echo needs_approval:true`) block until the user approves. Two approval channels:

- The Destreamed web UI / browser extension (existing)
- `/dt approve <echo-id>` shortcut inside Claude Code (NEW — bypasses the web UI for power users in flow)

## Approval flow

The `needs_approval` gate is the trust mechanism that makes "tuner working hand-in-hand with human" tractable. It is not a side-feature — it is the spine of every material decision in the loop. This section defines how it actually works end-to-end.

### Lifecycle

```
Tuner ─┐
       ├─► [1] PROPOSE  ───► add_echo(needs_approval: true) on task DESTREAM-81
       │                     body: { what, why, impact }
       │
       │     [2] SURFACE ───► echo visible on Destreamed (web, browser-ext)
       │                      AND printed inline in Claude Code chat with the echo-id
       │
       │     [3] DECIDE  ───► user picks one:
       │                      ├─ APPROVE  (channel A/B/C below)
       │                      ├─ DENY     (with optional reason)
       │                      └─ MODIFY   (deny + counter-proposal in the same beat)
       │
       │     [4] RESUME  ───► tuner reads the decision echo, then:
       │                      ├─ on APPROVE → execute the proposed step, log echo, continue loop
       │                      ├─ on DENY    → log a one-line "understood, pivoting to X" echo, ask user in chat or re-propose
       │                      └─ on MODIFY  → treat counter as the new direction; if still material, fresh proposal
       └◄──── (timeout: see "Stale approvals" below)
```

### Approval channels

| # | Channel | Where the user lives | Latency |
|---|---|---|---|
| A | Destreamed web UI | Browser tab on destreamed.com | Medium — tab switch needed |
| B | Browser extension | Existing webext | Low — toolbar click |
| C | `/dt approve <echo-id>` and `/dt deny <echo-id> [reason]` | Inside Claude Code chat | Lowest — no context switch |

Channel C is the **flow-preserving** one and the reason this gate doesn't kill productivity. When the tuner proposes, the chat output looks like:

```text
[tuner] Proposed plan as echo on DESTREAM-81 (needs_approval).
        Echo-ID: e_a1b2c3
          What:   rename `streams.name` → `streams.title` + migration 0042
          Why:    matches new vocabulary memo
          Impact: schema change + 14 frontend renames; reversible only via fresh migration

        Approve in chat:  /dt approve e_a1b2c3
        Deny:             /dt deny e_a1b2c3 [reason]
        Or use the Destreamed web UI / browser extension.

        Waiting...
```

### How the tuner waits

Hierarchy of preferences (server-capability-dependent):

1. **MCP server-sent events / streaming subscriptions** on `/echoes/:id/status` — tuner subscribes when proposing, wakes on the first status change. Best UX, requires server support.
2. **Polling** — every 5s for the first minute, then 30s for 5 min, then 2 min. Cheap fallback, fine for most cases.
3. **Manual nudge** — user types in chat "I approved it" → tuner re-checks immediately.

The tuner **does not stay idle**: while waiting, it can quietly do non-material prep work (read code, draft tests, write up the next sub-step) but **must not write anything that would be invalidated by a denial**.

### Decision outcomes — concrete behaviors

**APPROVE**
- Tuner records: who approved, at what timestamp, with what comment (if any).
- Executes the proposed step exactly as proposed. Deviation requires a fresh proposal.
- Logs a `add_echo` (no approval) with the result: "Done: <one-line>. Files: <list>." Continues loop.

**DENY**
- Tuner reads the denial reason verbatim.
- Logs a child-echo on the task: "Understood: <reason>. Pivoting to <one-line-alternative> — proposal incoming." or, if the denial is unclear, "Understood: <reason>. Need clarification: <question>." and waits for chat reply.
- Never silently retries the same proposal — that's the deny's purpose.

**MODIFY** (deny + counter)
- The denial echo's body contains a counter-proposal. Tuner treats the counter as the new direction.
- If the counter is still material (touches multiple files, irreversible, etc.), drops a fresh proposal echo with the counter as starting point. If trivial, executes directly with a status echo.

### Subagent approvals

A subagent that hits a `needs_approval` step has **two routing modes**, configured per-tuner in the tuner profile (`approval_routing` field, default: `direct-to-user`):

| Mode | Behavior |
|---|---|
| `direct-to-user` *(default)* | Subagent's proposal echo lands directly on the user — same channels A/B/C. Parent agent is notified but doesn't gate. **Trade-off:** more user interruptions, cleanest audit trail, no parent-AI exercising judgment over child-AI. |
| `parent-mediated` | Subagent's proposal is forwarded to the parent agent first. Parent can approve/deny/modify autonomously based on its system prompt. Only escalates to user if the parent itself is uncertain. **Trade-off:** fewer interruptions for the user, but parent now has authority that should arguably be human-only. Use sparingly, only for narrowly-scoped subagents (e.g., test-writer where the parent already has the spec). |

For v0.4.0 we ship `direct-to-user` as default and only expose `parent-mediated` to tuners that explicitly opt in via their server-side config. We monitor in beta whether anyone actually wants `parent-mediated`.

### Audit trail

Every proposal, approval, denial, and modification is an immutable echo on the task beat. The task's beat thread is the complete decision history — no out-of-band Slack messages, no "I told them in chat", no lost context. This is non-negotiable; **the audit trail is why the gate exists**, not a side-effect.

The tuner is forbidden from executing a material step whose proposal is not findable in the task's echo thread. If you can't `mcp__destreamed__beat get_echoes` and see "approve" on the relevant `needs_approval:true` echo, you don't act.

### Stale approvals

A proposal echo with `needs_approval:true` that sits unresolved for **24 hours** auto-expires server-side to `denied` with reason `"stale: no decision in 24h"`. On resume:

- The tuner reads the auto-deny.
- Posts a child-echo: "Proposal e_a1b2c3 expired. Re-proposing." or "Re-evaluating — context may have moved."
- Either re-proposes (if the work is still relevant) or asks the user.

The 24h window is server-configurable per stream via memo (e.g., a high-velocity stream can drop it to 4h; a slow legal-review stream can extend to 7d).

### What never gets gated

For UX sanity, these are explicitly **not** material and skip the gate:

- Reading files, running read-only commands (git status, ls, grep, MCP search calls).
- Drafting plans, summaries, explanations into the chat without writing.
- Dropping `kind: question` beats (asking the user is allowed without approval).
- Logging plain progress echoes (the per-step `add_echo` without `needs_approval`).

Gated, always:

- File writes/edits/deletes outside `/tmp` or scratch dirs.
- `git commit`, `git push`, branch deletes, force-pushes.
- Schema migrations, dependency upgrades, config changes.
- `drop_beat kind:memo` (proposing a new stream rule).
- `complete_task` (closing a task is itself a material decision).
- Any external API call with side effects (sending email, billing, etc.).

When in doubt: gate it. The gate is cheap (channel C is one chat line); the unwanted action is expensive.

## Architecture

### Client (this plugin)

**Slash commands**

| Command | Behavior |
|---|---|
| `/dt <ID>` | Switch focus to task with given ID |
| `/dt` | Interactive picker over my open tasks |
| `/dt subagent <desc>` | Spawn subagent for a sub-unit of the active task |
| `/dt approve <echo-id>` | Approve a pending `needs_approval` echo |
| `/dt search <kw>` | Prior-art lookup (replaces `/destreamed:search`) |
| `/dt answer <text>` | Reply to an open `kind:question` beat on the active task |

**Skills**

| Skill | Purpose |
|---|---|
| `destreamed-focus` | Orchestrates the switch — calls `task/:id/context`, primes Claude with tuner+memos+echoes |
| `destreamed-subagent` | Spawns Agent tool with the parent tuner profile baked into the system prompt |
| `destreamed-propose` | Submit plans/decisions/memos with `needs_approval: true` (unchanged from v0.3) |
| `destreamed-search` | Prior-art beat search (unchanged from v0.3) |

**Hooks**

| Hook | Purpose |
|---|---|
| `SessionStart` (startup\|clear) | One-line nudge: "Run `/dt <task-id>` or `/dt` to focus on a task before starting work." |
| `UserPromptSubmit` | Problem-keyword detection → search reminder (unchanged from v0.3) |

**MCP** unchanged (HTTP, OAuth via Dynamic Client Registration).

### Server (Phase E in destreamed-backend)

**New endpoints**

- `GET /api/v1/tuners/:slug` — full tuner profile
- `GET /api/v1/tasks/:id/context` — task context bundle (the JSON above)
- `GET /api/v1/tasks/me/open` — open tasks assigned to the authenticated user, lightweight (id, title, stream slug) for the `/dt` picker
- `POST /api/v1/echoes/:id/approve` — approve a pending echo
- `POST /api/v1/echoes/:id/deny` — deny with reason

**Schema additions**

- Table `tuners` (id, slug, display_name, system_prompt, allowed_tools[], default_memo_ids[], model_preference, created_at, updated_at)
- Column `tasks.assigned_tuner_id` (FK → tuners.id, nullable for legacy tasks)

**Auth/scopes**

OAuth scopes touched (existing or new):
- `tuners:read` (new) — read tuner profiles
- `tasks:read` already covered by `streams:read`?
- `echoes:approve` (new) — approve/deny pending echoes (separated from `beats:write` because approval has stronger semantics)

## Phase-E dependency

This entire spec assumes the server-side endpoints in §Server land. **Two paths forward:**

### Path 1 — Phase E ships first, plugin v0.4.0 follows

Cleanest. Plugin is a thin client over a real API. No interim hacks.

### Path 2 — Plugin ships interim, swaps to API in v0.5.0

If Phase E is more than ~2 weeks out, ship v0.4.0 with hardcoded fallbacks:

- **Tuner profile**: parse from a stream-level memo titled `tuner:<slug>` (body is the system prompt). Plugin hardcodes 2-3 default tuners (`backend-engineer`, `frontend-engineer`, `reviewer`) as YAML in the plugin repo.
- **Task context bundle**: assemble client-side from existing `mcp__destreamed__tasks` + `mcp__destreamed__stream` + `mcp__destreamed__beat` calls. Slower (3 round trips vs 1), but works.
- **Approval shortcut**: write echo with `kind: approval` linking the parent echo. Server-side gate logic stays in Destreamed.

**Recommendation**: Path 1. Path 2 is a hack-shape that will distort design choices.

## Open questions

1. **Tuner slug namespace** — global (`backend-engineer` is unique across all of Destreamed) or stream-scoped (`destreamed-backend/backend-engineer`)? Stream-scoped feels safer at scale; global is friendlier for sharing.
2. **Multi-tuner tasks** — can a task have multiple `assigned_tuner` entries (e.g., engineer + reviewer)? If yes, how does Claude pick which one to embody on `/dt`? (Suggestion: primary tuner = the one who acts; secondary tuners are read-only "advisors" whose memos are loaded but whose persona isn't installed.)
3. **Tool-whitelist enforcement** — Claude Code can't enforce `allowed_tools` at the runtime level (no MCP-level capability). It's advisory in the system prompt. Acceptable risk?
4. **Approval transport** — does `/dt approve` call a dedicated endpoint (`POST /api/v1/echoes/:id/approve`, semantically clean, audit-trail explicit) or drop a child-echo `kind:approval` (no schema change, but every consumer has to interpret echoes' meaning)? Recommendation: dedicated endpoint, mirrored as a child-echo for the audit trail. Best of both.
5. **Wait-channel** — does the MCP server already support server-sent events / subscriptions on echo status, or do we need polling for v0.4.0? Polling is fine for ship; SSE is much better UX. Worth a server-side estimate before locking the plugin pattern.
6. **Stale-approval grace period** — 24h server default sounds right; do we want per-task overrides (some tasks are urgent, some aren't), or only per-stream via memo? Per-task adds API surface; per-stream is enough for most cases.
7. **Approval routing default** — locked to `direct-to-user` for v0.4.0. Worth confirming once the first subagent-heavy use case ships (likely engineering-pair-style flow) whether `parent-mediated` ever earns its keep.
8. **Subagent context budget** — full tuner system prompt + all stream memos + task body + echoes might overflow a sub-agent's working context. Strategy: send tuner-prompt + memo titles only, fetch memo bodies on demand. Needs careful prompt template design.
9. **Deep-link from web** — `destreamed-claude://focus/DESTREAM-81` requires OS-level URL handler registration. Phase 2 feature; not blocking v0.4.0.
10. **Offline / disconnected behavior** — what happens to `/dt DESTREAM-81` when the network is down or OAuth has expired? Cache last-fetched context per task locally?

## Out of scope for v0.4.0

- Multi-task batching (work on 2+ tasks in parallel)
- Task creation from inside Claude Code (`/dt new "title"`) — keep it focus-only for now; user creates tasks in Destreamed
- Cross-stream task linking
- Notifications when an echo gets approved/denied while user is away
- Plugin-supplied default tuners (handled server-side per Path 1)
