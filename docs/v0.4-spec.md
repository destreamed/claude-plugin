# v0.4 Spec — Task-Focus Engine

> Status: **DRAFT**. Single source of truth for the v0.4 redesign. No code lands until this doc is signed off.

## TL;DR

A Claude Code session is no longer scoped to a project — it's scoped to a **Destreamed task**. One slash command (`/dt DESTREAM-81`) loads everything Claude needs to act competently in that task's context: the assigned tuner's persona, the stream's memos, the task's prior echoes, and any linked beats. Switching to another task in the same session swaps the entire context cleanly.

## Goals

1. **Task-first, not project-first.** Sessions begin by picking a task. Generic "open Claude in this folder" starts are no longer the default.
2. **Persona-as-tuner.** Each task has an `assigned_tuner`. Loading the task installs that tuner's persona — system-prompt layer, tool whitelist, default memos.
3. **Hot-swap.** `/dt <new-id>` mid-session replaces persona + memos + echoes-context. The active conversation history stays, but the *operating context* turns over.
4. **Subagents stay in role.** Spawned subagents inherit the parent's tuner profile + memos + task context, with a `needs_approval` gate on every material action.
5. **Sharing via Destreamed.** Tuners, memos, and tasks are server-side. Anyone with stream membership gets the same setup with no plugin update.

## Concepts

### Task ID format

`<STREAM-CODE>-<NUMBER>`, e.g., `DESTREAM-81`. Auto-generated by Destreamed when a task is created. Stream code is the uppercased prefix derived from the stream slug. This ID is the canonical handle for `/dt` switches and for deep-links.

### Tuner profile (server-side)

A tuner is a server-managed record:

| Field | Type | Purpose |
|---|---|---|
| `id` | uuid | Stable identifier |
| `slug` | string | Human reference (e.g., `backend-engineer`) |
| `display_name` | string | Shown in pickers and focus banner |
| `system_prompt` | string | The persona spec — added as a system-message layer when this tuner is active |
| `allowed_tools` | string[] | MCP/Claude tool whitelist (advisory; enforcement is via system prompt) |
| `default_memos` | uuid[] | Memos every task assigned to this tuner inherits |
| `model_preference` | enum? | `opus` / `sonnet` / `haiku` / null (Claude Code may override) |

When a task is fetched, its `assigned_tuner` field returns the full tuner profile inline.

### Task context bundle

One API call, one round trip, everything to switch focus:

```json
{
  "task": {
    "id": "DESTREAM-81",
    "title": "Fix race condition in stream-invite accept",
    "body": "...",
    "status": "in_progress",
    "stream_id": "..."
  },
  "stream": {
    "id": "...",
    "slug": "destreamed-backend",
    "name": "Destreamed Backend"
  },
  "tuner": {
    "slug": "backend-engineer",
    "display_name": "Backend Engineer",
    "system_prompt": "You are a senior Rust/Axum engineer...",
    "allowed_tools": ["read", "write", "bash:cargo", "mcp__destreamed__*"],
    "model_preference": "opus"
  },
  "memos": [
    { "id": "...", "title": "Rust style", "body": "..." },
    { "id": "...", "title": "No mocks in integration tests", "body": "..." }
  ],
  "echoes": [
    {
      "id": "...",
      "kind": "echo",
      "body": "Found root cause in services/streams.rs:142",
      "needs_approval": false,
      "approved": null,
      "created_at": "2026-04-29T..."
    }
  ],
  "linked": [
    { "id": "...", "kind": "memo", "title": "Stream invite spec" },
    { "id": "...", "kind": "question", "title": "Backfill or unique index?", "open": true }
  ]
}
```

### Subagent inheritance

When the main thread spawns a subagent (Claude Code's `Agent` tool), the prompt template is built from:

- The active tuner's `system_prompt` (verbatim)
- A condensed memo summary (titles + 1-line gist each, full bodies on request)
- The active task's title + body
- The specific sub-instruction from the parent
- A reminder line: "needs_approval: true is mandatory for any material change. Echo back to task DESTREAM-81 on completion."

The subagent's tool whitelist matches the parent tuner's `allowed_tools`. It can read everything, but writes (file edits, beat drops, completion) require a `needs_approval` gate.

## User experience

### Switching focus

```text
$ claude
> /dt DESTREAM-81
🎯 Switching focus → DESTREAM-81 — "Fix race condition in stream-invite accept"
   Tuner:  backend-engineer
   Stream: destreamed-backend
   Memos:  4 loaded (Rust style, error handling, no-mocks-in-tests, migration-naming)
   Echoes: 3 prior (last: "Found root cause in services/streams.rs:142")
   Linked: 1 spec memo, 1 open question

   ⚠ Open question on this task:
     "Do we want to backfill existing rows, or is the unique index enough?"
     → answer with `/dt answer <text>` or address it in your plan.

   Ready.
```

Without arguments, `/dt` opens an interactive picker over the user's open tasks (fuzzy-filterable by title and stream).

### Subagent

```text
> Plan was approved. Spawn a subagent to write the integration test for this fix
  while I refactor the handler.

[main] spawning subagent (tuner: backend-engineer, scope: write integration test)
       (subagent inherits memos, gets needs_approval gate on production writes)
```

When the subagent finishes, it posts a single echo on `DESTREAM-81` summarizing the work, and returns control.

### Approval

Material steps (`add_echo needs_approval:true`) block until the user approves. Two approval channels:

- The Destreamed web UI / browser extension (existing)
- `/dt approve <echo-id>` shortcut inside Claude Code (NEW — bypasses the web UI for power users in flow)

## Architecture

### Client (this plugin)

**Slash commands**

| Command | Behavior |
|---|---|
| `/dt <ID>` | Switch focus to task with given ID |
| `/dt` | Interactive picker over my open tasks |
| `/dt subagent <desc>` | Spawn subagent for a sub-unit of the active task |
| `/dt approve <echo-id>` | Approve a pending `needs_approval` echo |
| `/dt search <kw>` | Prior-art lookup (replaces `/destreamed:search`) |
| `/dt answer <text>` | Reply to an open `kind:question` beat on the active task |

**Skills**

| Skill | Purpose |
|---|---|
| `destreamed-focus` | Orchestrates the switch — calls `task/:id/context`, primes Claude with tuner+memos+echoes |
| `destreamed-subagent` | Spawns Agent tool with the parent tuner profile baked into the system prompt |
| `destreamed-propose` | Submit plans/decisions/memos with `needs_approval: true` (unchanged from v0.3) |
| `destreamed-search` | Prior-art beat search (unchanged from v0.3) |

**Hooks**

| Hook | Purpose |
|---|---|
| `SessionStart` (startup\|clear) | One-line nudge: "Run `/dt <task-id>` or `/dt` to focus on a task before starting work." |
| `UserPromptSubmit` | Problem-keyword detection → search reminder (unchanged from v0.3) |

**MCP** unchanged (HTTP, OAuth via Dynamic Client Registration).

### Server (Phase E in destreamed-backend)

**New endpoints**

- `GET /api/v1/tuners/:slug` — full tuner profile
- `GET /api/v1/tasks/:id/context` — task context bundle (the JSON above)
- `GET /api/v1/tasks/me/open` — open tasks assigned to the authenticated user, lightweight (id, title, stream slug) for the `/dt` picker
- `POST /api/v1/echoes/:id/approve` — approve a pending echo
- `POST /api/v1/echoes/:id/deny` — deny with reason

**Schema additions**

- Table `tuners` (id, slug, display_name, system_prompt, allowed_tools[], default_memo_ids[], model_preference, created_at, updated_at)
- Column `tasks.assigned_tuner_id` (FK → tuners.id, nullable for legacy tasks)

**Auth/scopes**

OAuth scopes touched (existing or new):
- `tuners:read` (new) — read tuner profiles
- `tasks:read` already covered by `streams:read`?
- `echoes:approve` (new) — approve/deny pending echoes (separated from `beats:write` because approval has stronger semantics)

## Phase-E dependency

This entire spec assumes the server-side endpoints in §Server land. **Two paths forward:**

### Path 1 — Phase E ships first, plugin v0.4.0 follows

Cleanest. Plugin is a thin client over a real API. No interim hacks.

### Path 2 — Plugin ships interim, swaps to API in v0.5.0

If Phase E is more than ~2 weeks out, ship v0.4.0 with hardcoded fallbacks:

- **Tuner profile**: parse from a stream-level memo titled `tuner:<slug>` (body is the system prompt). Plugin hardcodes 2-3 default tuners (`backend-engineer`, `frontend-engineer`, `reviewer`) as YAML in the plugin repo.
- **Task context bundle**: assemble client-side from existing `mcp__destreamed__tasks` + `mcp__destreamed__stream` + `mcp__destreamed__beat` calls. Slower (3 round trips vs 1), but works.
- **Approval shortcut**: write echo with `kind: approval` linking the parent echo. Server-side gate logic stays in Destreamed.

**Recommendation**: Path 1. Path 2 is a hack-shape that will distort design choices.

## Open questions

1. **Tuner slug namespace** — global (`backend-engineer` is unique across all of Destreamed) or stream-scoped (`destreamed-backend/backend-engineer`)? Stream-scoped feels safer at scale; global is friendlier for sharing.
2. **Multi-tuner tasks** — can a task have multiple `assigned_tuner` entries (e.g., engineer + reviewer)? If yes, how does Claude pick which one to embody on `/dt`? (Suggestion: primary tuner = the one who acts; secondary tuners are read-only "advisors" whose memos are loaded but whose persona isn't installed.)
3. **Tool-whitelist enforcement** — Claude Code can't enforce `allowed_tools` at the runtime level (no MCP-level capability). It's advisory in the system prompt. Acceptable risk?
4. **Echo-approval channel** — is `/dt approve` a Destreamed-API call (POST `/echoes/:id/approve`), or does it drop a child-echo with `kind:approval` (current convention)? The former is cleaner; the latter requires no schema change.
5. **Subagent context budget** — full tuner system prompt + all stream memos + task body + echoes might overflow a sub-agent's working context. Strategy: send tuner-prompt + memo titles only, fetch memo bodies on demand. Needs careful prompt template design.
6. **Deep-link from web** — `destreamed-claude://focus/DESTREAM-81` requires OS-level URL handler registration. Phase 2 feature; not blocking v0.4.0.
7. **Offline / disconnected behavior** — what happens to `/dt DESTREAM-81` when the network is down or OAuth has expired? Cache last-fetched context per task locally?

## Out of scope for v0.4.0

- Multi-task batching (work on 2+ tasks in parallel)
- Task creation from inside Claude Code (`/dt new "title"`) — keep it focus-only for now; user creates tasks in Destreamed
- Cross-stream task linking
- Notifications when an echo gets approved/denied while user is away
- Plugin-supplied default tuners (handled server-side per Path 1)
